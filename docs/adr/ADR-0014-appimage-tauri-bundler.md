# ADR-0014 — AppImage distribution via Tauri's native bundler

**Date:** 2026-04-04
**Status:** Accepted

## Context

FS-DIST requires TauTerm v1 to be distributed as an AppImage (FS-DIST-001), self-contained except for standard Linux desktop runtime dependencies (FS-DIST-002), executable directly after `chmod +x` (FS-DIST-004), and available for five architectures: x86 (i686), x86_64, ARM32 (armhf), ARM64 (aarch64), RISC-V (riscv64) (FS-DIST-003).

TauTerm is built with Tauri 2. The Tauri framework includes a native AppImage bundler that produces a valid AppImage without requiring external tools (`appimagetool`, `linuxdeploy`, `pkg2appimage`). The bundler is invoked by `pnpm tauri build` when `"appimage"` is included in the `bundle.targets` array in `tauri.conf.json`.

## Decision

Use **Tauri's native AppImage bundler** exclusively for producing TauTerm's AppImage distribution artefacts.

No external AppImage toolchain (`appimagetool`, `linuxdeploy`) is introduced. The bundler is configured via `tauri.conf.json`:

```json
{
  "bundle": {
    "targets": ["appimage"],
    "identifier": "dev.tauterm.app",
    "icon": ["icons/32x32.png", "icons/128x128.png", "icons/128x128@2x.png", "icons/icon.icns", "icons/icon.ico"]
  }
}
```

The `.desktop` entry and application icon required by FS-DIST-005 are generated by the Tauri bundler from the bundle configuration (identifier, icon, and product name fields).

**Runtime dependency — WebKitGTK:** The AppImage bundles the Rust binary and all frontend assets but does not bundle WebKitGTK (`libwebkit2gtk-4.1` or `libwebkit2gtk-4.0`). This is a deliberate trade-off: bundling WebKitGTK would add ~80–150 MB to the AppImage size and would require maintaining a pinned WebKitGTK version inside the bundle. WebKitGTK is a standard component of any GNOME-compatible Linux desktop environment. The FS-DIST-002 requirement ("self-contained except for a standard Linux desktop environment display server and GTK/equivalent WebView runtime") explicitly accommodates this dependency.

**Multi-architecture strategy:** Each target architecture is built in a separate CI job on a native or QEMU-emulated runner. Cross-compilation is avoided: Tauri's AppImage bundler requires WebKitGTK headers and libraries matching the target architecture at build time, which makes cross-compilation impractical without a matching sysroot. Native or QEMU-emulated builds are the reliable path. See ARCHITECTURE.md §10.6 for the full per-architecture target triple table.

## Alternatives considered

**External appimagetool + linuxdeploy pipeline**
`linuxdeploy` with the GTK plugin can automatically bundle shared libraries (excluding WebKitGTK, which is excluded by policy). This gives more control over the AppImage contents but requires maintaining a separate build pipeline outside of Tauri. For TauTerm, where Tauri already owns the build pipeline, adding an external tool introduces unnecessary complexity and a second dependency to version. Rejected: Tauri's native bundler is sufficient and simpler.

**Flatpak distribution** *(deferred, not rejected)*
Flatpak is a strong candidate for post-v1 distribution: sandboxed, excellent desktop integration, and a centralized repository via Flathub. However, it introduces non-trivial complexity — a runtime manifest, a separate build system (`flatpak-builder`), and a Flathub submission and review process that requires a certain project maturity. For the MVP, this overhead is not justified. Deferred to a future release once the project has stabilised.

**`.deb` / `.rpm` / distribution-native packages** *(deferred, not rejected)*
Tauri also produces `.deb` and `.rpm` packages natively. These are appropriate for system-wide installation and package manager integration and are a natural progression for wider Linux distribution coverage. However, each format carries distribution-specific packaging conventions, signing requirements, and repository submission processes. This complexity is better addressed after the MVP. Deferred to a future release.

## Consequences

**Positive:**
- Single build command (`pnpm tauri build`) produces the AppImage artefact with no external toolchain required.
- `.desktop` entry and icon are generated automatically by the Tauri bundler (FS-DIST-005).
- Consistent with the existing Tauri-managed build pipeline (ARCHITECTURE.md §10.1).
- AppImage is executable directly after `chmod +x` (FS-DIST-004).

**Negative / risks:**
- WebKitGTK is a runtime dependency. On a minimal Linux installation without a desktop environment (e.g., a headless server), TauTerm will not run. This is an inherent constraint of the Tauri/WebView architecture (ADR-0001) and is accepted.
- Multi-architecture builds require separate CI jobs or QEMU emulation. ARM32 and RISC-V builds will be slower than x86_64 due to emulation overhead. This is a CI cost, not a user-facing issue.
- Tauri's AppImage bundler does not support AppImage Type 2 update channels. This is acceptable for v1; update delivery is out of scope.

## Notes

The AppImage artefact naming convention is `TauTerm-{version}-{arch}.AppImage` (e.g., `TauTerm-1.0.0-x86_64.AppImage`). This matches the convention used by other Tauri-based applications and is recognizable to users.

WebKitGTK version compatibility: `libwebkit2gtk-4.1` is the preferred variant (used by Tauri 2 on Ubuntu 22.04+). On older distributions, `libwebkit2gtk-4.0` may be required. The Tauri build system selects the appropriate variant based on the build environment. This is transparent to the user.
