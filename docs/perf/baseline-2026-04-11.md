## Key observations

1. **serde_json est le goulot d'étranglement IPC** : 1.53 ms pour sérialiser un frame 220×50 — soit ~13% du budget de 12 ms par frame. Candidat prioritaire pour P5 (flat binary buffer / compact repr).

2. **`read_under_writer` à 777 µs** : la Task 2 (émission) peut être bloquée ~1 frame entier si Task 1 tient le write-lock. Contention RwLock significative à surveiller.

3. **Full redraw 698 µs vs partial 415 µs** : la sérialisation JSON (1.53 ms) domine les deux — réduire les allocations String par cellule aurait plus d'impact que le chemin full vs partial.

4. **VT réaliste plus rapide que ASCII pur** (33 vs 21 MiB/s) : les séquences CSI courtes ont un overhead par-octet inférieur aux 220×50 passes cellule-par-cellule du pur ASCII.

5. **`dirty_region merge` à 95 ns et `dirty_rows` à ~200 ps** : les primitives de suivi sont quasi-gratuites — pas de gain à chercher là.

6. **vtebench ~8 ms** : la latence CPR mesurée depuis l'intérieur du terminal inclut le round-trip Rust PTY read → VT process → write back. Referencing point pour futurs comparatifs avec Alacritty/foot.

## Post-optimisation — P-IPC1 (2026-04-11)

**Optimisation** : `#[serde(default, skip_serializing_if = "is_false")]` sur les 7 champs booléens de `CellAttrsDto` + `#[serde(default, skip_serializing_if = "is_zero")]` sur `underline: u8`. Les champs à valeur par défaut sont omis du JSON.

| Benchmark | Avant | Après | Δ |
|---|---|---|---|
| `serde_json_ipc/serialize_full_redraw_220x50` | 1.53 ms | **667 µs** | **−56 %** |
| `default_attrs/serialize_full_redraw_220x50_default_attrs` (nouveau) | — | **658 µs** | — |

Le bench existant utilisait déjà des cellules à attrs par défaut (bold=false, etc.) — c'est le cas typique du contenu terminal. La réduction de 56 % provient de l'élimination de 8 champs JSON par cellule × 11 000 cellules = ~88 000 tokens JSON supprimés par frame.

**TypeScript** : champs `bold?`, `dim?`, `italic?`, `underline?`, `blink?`, `inverse?`, `hidden?`, `strikethrough?` rendus optionnels dans `CellAttrsDto`. Frontend lit ces champs avec `?? false` / `?? 0`.

## Post-optimisation — P-IPC2 (2026-04-11)

**Optimisation** : debounce adaptatif dans Task 2. Ajout de `needs_immediate_flush: bool` à `ProcessOutput`. Task 1 set ce flag quand des réponses VT (CPR, DA, DSR) ont été générées. Task 2 bypasse le debounce de 12 ms et flush immédiatement.

| Métrique | Avant | Après | Δ | Notes |
|---|---|---|---|---|
| vtebench `dense_cells` CPR round-trip avg | ~8 ms | ~9 ms | ~+1 ms (bruit) | Voir explication ci-dessous |
| `screen-update` event latency après CPR | 0–12 ms (avg 6 ms) | ~0 ms (immédiat) | **−6 ms avg** | Latence non mesurable par vtebench |

**Pourquoi vtebench ne mesure pas l'amélioration :**  
vtebench mesure le round-trip PTY-level : `ESC[6n` → `ESC[row;colR` lus depuis le PTY master. Cette réponse est écrite par Task 1 IMMÉDIATEMENT (avant même que le `ProcessOutput` soit envoyé à Task 2), donc indépendante du debounce. P-IPC2 réduit la latence de l'événement `screen-update` émis vers le frontend — ce que les apps comme `vim`/`neovim` attendent pour savoir où se trouve le curseur dans leur rendu. Cette latence n'est pas capturée par vtebench ; elle nécessiterait une instrumentation frontend.

## Post-optimisation — P12a (cell-level dirty tracking)

**Optimisation :** Remplacement du dirty tracking au niveau de la *ligne* (WP3c) par des mutations au niveau de la *cellule*. Avant P12a, chaque ligne sale déclenchait `gridRows[r] = Array.from(...)` — un nouveau tableau proxy remplaçait la ligne, forçant Svelte à réévaluer les 220 expressions de cellule. Après P12a, `gridRows[r][c] = grid[...]` — le proxy profond de Svelte 5 n'invalide que le span de la cellule modifiée.

Instrumentation ajoutée : `performance.mark('tauterm:asu:start')` au début de `applyScreenUpdate()`, `$effect` post-rendu qui émet `performance.measure('tauterm:frameRender', ...)`. Mesurable via DevTools Console :

```javascript
// Avec htop actif pendant 5 s :
setTimeout(() => {
  const e = performance.getEntriesByName('tauterm:frameRender').slice(-100);
  const avg = e.reduce((s, x) => s + x.duration, 0) / e.length;
  console.log('avg frameRender ms:', avg.toFixed(2));
}, 5000);
```

**Méthodologie :** Benchmark E2E WebdriverIO (`tests/e2e/perf-p12a-frame-render.spec.ts`). Mesures via `performance.getEntriesByName('tauterm:frameRender')` lues depuis le WebView WebKitGTK par `browser.execute()`. Deux workloads :
- **SCROLL** : 30 batches × 12 lignes colorisées (SGR), 20 ms entre batches → simule log tailing
- **CURSOR** : 60 mises à jour sparse (4 cellules par update via `ESC[row;colH`) → simule ncurses/htop

**Résultats mesurés :**

| Workload | Avant P12a (WP3c) | Après P12a | Δ avg | Δ p95 |
|---|---|---|---|---|
| SCROLL — avg `tauterm:frameRender` | 23.53 ms | 23.07 ms | **−2 %** | −0 % |
| SCROLL — p95 | 28.00 ms | 29.00 ms | +4 % (bruit) | — |
| CURSOR — avg `tauterm:frameRender` | 1.22 ms | 1.15 ms | **−6 %** | −0 % |
| CURSOR — p95 | 2.00 ms | 2.00 ms | 0 % | — |
| IDLE — renders spurieux | 0 | 0 | ✓ | — |

**Interprétation :**

P12a apporte une amélioration mesurable mais faible (−2 % scroll, −6 % cursor) sur ce matériel. L'écart est dans le bruit de mesure pour le workload scroll (la résolution de `performance.measure` sous WebKitGTK est ~1 ms).

**Diagnostic du vrai goulot :** Le workload SCROLL à 23 ms moyen dépasse le budget de 12 ms par frame. P12a ne résout pas ce problème parce que le goulot n'est **pas** dans l'itération JS des cellules — il est dans les mutations DOM elles-mêmes (WebKitGTK repaint) et/ou dans le pipeline IPC (désérialisation + dispatch d'événements). Le workload CURSOR à 1.15 ms confirme que les mises à jour sparse sont déjà rapides.

**Décision P12b :** Le workload scroll dépasse le budget de frame. P12b (WebGL2) n'est pas la prochaine étape recommandée — les contraintes WebKitGTK (transparence incompatible, sélection à réimplémenter, AT-SPI2 parallèle) sont importantes. Investiguer d'abord le bottleneck réel : `performance.measure('tauterm:applyScreenUpdate', ...)` pour isoler la part JS, vs profiler les repaints WebKitGTK.

## Post-diagnostic — P-DIAG-1 (applyOnly vs repaintTime decomposition, 2026-04-11)

**Objectif :** Quantifier la part JS pure (`applyOnly`) vs la part Svelte reconcile + WebKitGTK repaint (`repaintTime`) dans chaque frame. Instrumentation : `tauterm:asu:start` → `tauterm:apply:end` → `tauterm:render:end`. Activée via `VITE_PERF_INSTRUMENTATION=1` (flag dédié ajouté pour éviter de modifier le comportement des builds normaux).

**Méthodologie :** Benchmark E2E WebdriverIO (`tests/e2e/perf-p12a-frame-render.spec.ts`), build `VITE_PERF_INSTRUMENTATION=1 pnpm tauri build --no-bundle -- --features e2e-testing`.

| Workload | applyOnly avg | repaintTime avg | frameRender avg | repaint% |
|---|---|---|---|---|
| SCROLL 30×12 lignes | 4.90 ms | 17.87 ms | 22.77 ms | **78 %** |
| CURSOR 60 sparse | 0.22 ms | 0.98 ms | 1.20 ms | **82 %** |

**Interprétation :**

- La part JS (`applyOnly`) représente 18–22 % du temps total. Le goulot est bien dans WebKitGTK repaint (78–82 %), pas dans l'itération des cellules.
- P12a (cell-level dirty) n'avait qu'un impact de 2–6 % parce qu'il optimisait la partie JS déjà minoritaire.
- Décision : P-OPT-1 (rAF batching) est la prochaine optimisation prioritaire — coalescer N événements `screen-update` arrivant entre deux frames en 1 seul repaint WebKitGTK.

## Post-optimisation — P-OPT-1 (rAF batching of screen-update events, 2026-04-11)

**Optimisation :** Remplacement de `applyScreenUpdate(update)` direct dans le listener `screen-update` par une file `rafQueue` + `requestAnimationFrame(flushRafQueue)`. N événements arrivant entre deux frames sont appliqués séquentiellement dans le callback rAF (ordre préservé — chaque événement est un delta depuis le précédent, pas un snapshot complet). Svelte 5 batche les mutations `$state` synchrones → 1 cycle DOM + 1 repaint WebKitGTK au lieu de N.

**Implémentation :**
- `rafScheduled` boolean séparé de `rafId` pour éviter une race où un rAF synchrone (tests) efface `rafId` avant l'assignation.
- La logique de buffering WP3b (pendant le fetch du snapshot initial) est inchangée — les events pendant l'init restent dans `pendingUpdates` et sont rejoués synchronement.
- `onDestroy` appelle `cancelAnimationFrame(rafId)` si un frame est pending.

**Résultats mesurés :**

| Workload | Avant P-OPT-1 | Après P-OPT-1 | Δ avg | Notes |
|---|---|---|---|---|
| SCROLL avg `tauterm:frameRender` | 23.07 ms | 22.77 ms | −1.3 % | Dans le bruit — `INTER_BATCH_MS=20ms > 16ms` rAF, events rarement coalesced |
| SCROLL p95 | 29.00 ms | 28.00 ms | −3.4 % | — |
| CURSOR avg | 1.15 ms | 1.20 ms | +4 % (bruit) | — |
| RAPID-FIRE (5 events @ 3 ms) | n/a (non mesuré avant) | 3 entrées `frameRender` | **−40 % repaints** | 5 events → 3 frames (coalescing actif) |

**Interprétation :**

- P-OPT-1 n'améliore pas significativement le workload SCROLL car les batches arrivent espacées de 20 ms (> 16 ms rAF window) — chaque batch génère déjà 1 seul frame.
- L'impact est mesurable sur les bursts haute fréquence : RAPID-FIRE (5 events à 3 ms d'espacement) produit 3 `frameRender` au lieu de 5 attendus — 40 % de repaints WebKitGTK évités pour les mises à jour rapides (ncurses, htop, vim scroll).
- Le vrai bottleneck reste le repaint WebKitGTK (78 % du temps). La prochaine optimisation significative nécessiterait de réduire la surface de repaint (CSS `contain`, `will-change`) ou de passer à un renderer canvas.

## Post-optimisation — P-OPT-3 (CSS contain: strict, 2026-04-11)

**Optimisation :** Ajout de `contain: strict` sur `.terminal-pane__viewport`. Équivalent à `contain: size layout style paint` — confine les invalidations layout et paint à la sous-arborescence du viewport. WebKitGTK ne repeint que dans le bounding box du viewport au lieu de remonter dans l'arbre de rendu.

**Implémentation :** Une ligne CSS dans `:global(.terminal-pane__viewport)` dans `src/lib/components/TerminalPane.svelte`. `contain: size` applicable car le viewport a `width: 100%; height: 100%` — taille CSS explicite. Le cursor overlay (`position: absolute`) reste bien positionné dans le viewport car `contain: layout` maintient l'élément comme containing block pour ses descendants absolus.

Pas de repli sur `contain: layout paint` nécessaire — aucun artefact visuel observé.

| Workload | Avant P-OPT-3 | Après P-OPT-3 | Δ avg | Δ repaintTime | Δ repaint% |
|---|---|---|---|---|---|
| SCROLL — frameRender avg | 27.54 ms | **23.33 ms** | **−15.3 %** | 21.43→18.13 ms (−15.4 %) | 78 %→78 % |
| SCROLL — frameRender p95 | 35.00 ms | **28.00 ms** | **−20.0 %** | — | — |
| SCROLL — applyOnly avg | 6.11 ms | 5.20 ms | −14.9 % | — | — |
| CURSOR — frameRender avg | 1.27 ms | **1.17 ms** | **−7.9 %** | 0.98→0.92 ms (−6.1 %) | 78 %→79 % |
| CURSOR — frameRender p95 | 2.00 ms | 2.00 ms | 0 % | — | — |

**Interprétation :**

- La réduction de repaintTime SCROLL (−15.4 %, soit −3.30 ms) confirme que `contain: paint` réduit effectivement la surface de repaint WebKitGTK sur ce workload.
- Le p95 SCROLL passe de 35→28 ms (−20 %) — l'effet est plus marqué sur les pics que sur la moyenne, ce qui est cohérent avec une réduction des repaints "débordants" dans l'arbre de rendu lors des rafales.
- applyOnly SCROLL baisse aussi (6.11→5.20 ms, −14.9 %) — inattendu, probablement du bruit inter-session plutôt qu'un effet direct de `contain: layout`.
- CURSOR : amélioration marginale (−7.9 %) — workload déjà rapide, variabilité de mesure à 1 ms de résolution.
- Le workload SCROLL reste au-dessus du budget de 12 ms (23.33 ms). P-OPT-3 réduit l'écart mais ne comble pas le déficit. P-OPT-4 (`will-change` sur les rows) est la prochaine étape.

## Post-optimisation — P-OPT-4 (will-change: contents sur .terminal-pane__row, 2026-04-11)

**Optimisation :** Ajout de `will-change: contents` sur chaque `.terminal-pane__row`. L'intention est de promouvoir chaque row dans un compositing layer indépendant dans WebKitGTK, permettant au moteur de repeindre uniquement les rows dont les cellules ont changé sans invalider la grille entière. Approche statique (24 rows, ~50 MB GPU max sur les systèmes cibles).

**Implémentation :** Une ligne CSS dans `:global(.terminal-pane__row)` dans `src/lib/components/TerminalPane.svelte`.

| Workload | Avant P-OPT-4 (post-P-OPT-3) | Après P-OPT-4 | Δ avg | Δ repaintTime |
|---|---|---|---|---|
| SCROLL — frameRender avg | 23.33 ms | **23.23 ms** | **−0.4 %** (bruit) | 18.13→18.03 ms (−0.6 %) |
| SCROLL — frameRender p95 | 28.00 ms | **29.00 ms** | +3.6 % (bruit) | — |
| SCROLL — applyOnly avg | 5.20 ms | 5.20 ms | 0 % | — |
| CURSOR — frameRender avg | 1.17 ms | **1.22 ms** | +4.3 % (bruit) | 0.92→0.98 ms (bruit) |
| CURSOR — frameRender p95 | 2.00 ms | 2.00 ms | 0 % | — |

**Interprétation :**

- `will-change: contents` n'a **pas d'effet mesurable** dans WebKitGTK. Les deltas SCROLL (−0.4 % avg, +3.6 % p95) et CURSOR (+4.3 % avg) sont entièrement dans le bruit de mesure inter-run.
- Ce résultat est cohérent avec la mise en garde formulée avant l'implémentation : `will-change: contents` est dans la spec CSS mais son comportement sous WebKitGTK (le WebView de Tauri Linux) est incertain. Le moteur ne semble pas créer de compositing layers par row pour cette valeur, ou le coût de synchronisation des layers annule tout gain.
- La propriété est conservée (pas de régression, coût CSS négligeable) mais ne contribue pas à l'amélioration des performances.

**Recommandation pour la prochaine étape :**

Tester `will-change: transform` (ou `transform: translateZ(0)`) comme alternative — ces valeurs sont connues pour forcer la promotion en compositing layer dans les moteurs WebKit/Blink indépendamment de `contents`. À tester dans une étape P-OPT-5 dédiée avec le même benchmark. Si WebKitGTK ne supporte pas non plus cette promotion par row, le levier suivant est `transform: translateZ(0)` appliqué dynamiquement uniquement aux rows marquées dirty (promotion à la volée via JS, supprimée après repaint).
