## How to use this document

The test scenarios here are organized by implementation concern, not by feature. They are intended to be consumed by:

- **rust-dev** building `vt/mouse.rs`, `vt/osc.rs`, `vt/search.rs` — use the byte tables and `VtTestCase` fixtures directly.
- **frontend-dev** building `terminal/keyboard.ts`, `terminal/mouse.ts` — use the encoding tables and edge-case matrix.
- **test-engineer** registering these cases in `src-tauri/tests/vt_conformance.rs` — copy the `VtTestCase` structs into the data-driven runner.

Security scenarios are excluded — see `security-pty-ipc-ssh-credentials-csp-osc52.md`.

## Part 1: VT Domain — Byte-Level Specifications

### 1.1 Mouse Reporting (FS-VT-080 to 086)

#### Standard references

- VT220 Programmer Reference Manual §4.1, §5.4
- xterm `ctlseqs.ms` §Mouse Tracking — the normative reference for all four reporting modes and three encodings
- ECMA-48 is silent on mouse; xterm extensions are the de facto standard

#### Mode activation sequences

| Mode name | DECSET sequence | DECRST sequence | FS ref |
|---|---|---|---|
| X10 (click only) | `ESC [ ? 9 h` | `ESC [ ? 9 l` | FS-VT-080 |
| Normal (press+release) | `ESC [ ? 1000 h` | `ESC [ ? 1000 l` | FS-VT-080 |
| Button-event (motion while pressed) | `ESC [ ? 1002 h` | `ESC [ ? 1002 l` | FS-VT-080 |
| Any-event (all motion) | `ESC [ ? 1003 h` | `ESC [ ? 1003 l` | FS-VT-080 |
| SGR extended encoding | `ESC [ ? 1006 h` | `ESC [ ? 1006 l` | FS-VT-081 |
| URXVT extended encoding | `ESC [ ? 1015 h` | `ESC [ ? 1015 l` | FS-VT-081 |
| Focus events | `ESC [ ? 1004 h` | `ESC [ ? 1004 l` | FS-VT-084 |

**Mode priority:** When both mode 1002 (ButtonEvent) and 1003 (AnyEvent) are active simultaneously, AnyEvent takes precedence. When SGR (1006) and URXVT (1015) are both active, SGR takes precedence (ARCHITECTURE.md §14.5, `mouse.ts` detail).

#### X10 encoding (default, modes 9/1000/1002/1003)

```
ESC [ M <Cb> <Cx> <Cy>
```

- `Cb` = button byte = `32 + button_code + modifier_bits`
- `Cx` = `32 + col` (1-indexed)
- `Cy` = `32 + row` (1-indexed)

Button codes:
| Event | button_code |
|---|---|
| Left press | 0 |
| Middle press | 1 |
| Right press | 2 |
| Release (any button) | 3 |
| Scroll up | 64 |
| Scroll down | 65 |

Modifier bits (additive):
| Modifier | bit value |
|---|---|
| Shift | 4 |
| Alt/Meta | 8 |
| Ctrl | 16 |
| Motion flag (ButtonEvent/AnyEvent) | 32 |

**Overflow constraint:** `Cx` and `Cy` are single bytes. `32 + 223 = 255` (max valid). Columns or rows above 223 produce byte overflow and are unrepresentable in X10. The encoder MUST NOT emit overflowed bytes — either clamp to 223 or refuse to encode if SGR is not available.

#### SGR encoding (mode 1006)

```
press:   ESC [ < <Cb> ; <Cx> ; <Cy> M
release: ESC [ < <Cb> ; <Cx> ; <Cy> m
```

- `Cb` = button code (decimal, same table as X10 but without the +32 offset)
- `Cx`, `Cy` = decimal integers (1-indexed, unbounded)
- Lowercase `m` marks release; uppercase `M` marks press

**Key difference from X10:** Release events carry the actual button that was released, not the sentinel value 3. Middle-button release = `ESC [ < 1 ; cx ; cy m`.

#### URXVT encoding (mode 1015)

```
ESC [ <Cb> ; <Cx> ; <Cy> M
```

- Same as SGR but no `<` prefix and release is always encoded with button code `3`
- No trailing `m` vs `M` distinction

#### Focus events (mode 1004)

```
focus-in:  ESC [ I   (0x1B 0x5B 0x49)
focus-out: ESC [ O   (0x1B 0x5B 0x4F)
```

Generated by the **frontend** when the pane element receives or loses focus (not by the VtProcessor). Requires the `mode-state-changed` event to carry `focusEvents: bool`.

#### Application compatibility matrix

| Application | Mode(s) used | Encoding | What breaks if not implemented |
|---|---|---|---|
| vim (`set mouse=a`) | 1000, 1002 | X10 then SGR if available | Click-to-position, visual select, scroll |

**Shift+Click bypass (FS-VT-083):** Must be checked before any mouse mode routing. The frontend checks `event.shiftKey` first; if true, the event is handled locally (selection) regardless of active PTY mouse mode. This is an xterm convention that allows the user to always access terminal-side selection even when an application has captured the mouse.

### 1.2 Bracketed Paste (FS-CLIP-008)

#### Standard reference

xterm `ctlseqs.ms` §Bracketed Paste Mode. No ECMA/ISO normative reference; xterm de facto standard.

#### Protocol

```
DECSET:   ESC [ ? 2004 h    (enable bracketed paste)
DECRST:   ESC [ ? 2004 l    (disable bracketed paste)

Paste begin marker: ESC [ 2 0 0 ~   (bytes: 0x1B 0x5B 0x32 0x30 0x30 0x7E)
Paste end marker:   ESC [ 2 0 1 ~   (bytes: 0x1B 0x5B 0x32 0x30 0x31 0x7E)
```

The pasted text is delivered verbatim between the two markers. The application (shell, editor) sees the markers as delimiters and does not interpret escape sequences within the pasted region.

#### Injection prevention (mandatory)

Any occurrence of the paste-end marker `ESC [201~` within the pasted content MUST be stripped before wrapping. Failure to do so allows a crafted clipboard value to terminate the bracketed region early and inject text that the shell processes as if typed.

Implementation: scan pasted bytes for `\x1b[201~` (6 bytes: `1B 5B 32 30 31 7E`) and remove each occurrence before constructing the wrapped payload.

#### State tracking in the frontend

The frontend must track whether bracketed paste mode is currently active. This state is communicated via the `mode-state-changed` event. The current architecture payload carries `{ paneId, decckm, deckpam }` — it must be extended with `bracketedPaste: bool` to enable the frontend paste handler to know which path to take.

Alternative: the frontend can track mode 2004 independently by parsing DECSET/DECRST events in the `screen-update` stream, but this duplicates VT parsing in the frontend and is not the preferred approach.

#### `VtTestCase` fixtures

These fixtures test the VtProcessor-side mode tracking only. The paste wrapping logic is frontend-side and belongs in vitest.

| Test name | Input bytes | Expected mode state |
|---|---|---|
| `FS-CLIP-008-decset-2004` | `\x1b[?2004h` | `bracketedPaste: true` |
| `FS-CLIP-008-decrst-2004` | `\x1b[?2004h` then `\x1b[?2004l` | `bracketedPaste: false` |
| `FS-CLIP-008-alt-screen-reset` | `\x1b[?2004h` then `\x1b[?1049h` | alternate screen: bracketedPaste reset to false |
| `FS-CLIP-008-alt-screen-restore` | `\x1b[?2004h`, switch alt, switch back | normal screen: bracketedPaste restored to true |

The last two cases verify ARCHITECTURE.md §5.3: on transition to alternate screen (mode 1049), the entire mode state is saved and reset to defaults. On return, the saved state is restored.

### 1.3 OSC Title Sequences (FS-VT-060 to 062, FS-TAB-006)

#### Standard reference

ECMA-48 §8.3.89 (OSC — Operating System Command). xterm `ctlseqs.ms` §Operating System Controls. The OSC terminator is BEL (0x07) or ST (`ESC \` = 0x1B 0x5C); both must be accepted.

#### OSC title sequences

| Sequence | Effect | FS ref |
|---|---|---|
| `OSC 0 ; title BEL/ST` | Set icon name AND window title | FS-VT-060 |
| `OSC 1 ; title BEL/ST` | Set icon name only — MAY be ignored | FS-VT-060 |
| `OSC 2 ; title BEL/ST` | Set window title only | FS-VT-060 |
| `OSC 22 ; title BEL/ST` | Push title onto stack | FS-VT-061 |
| `OSC 23 ; title BEL/ST` | Pop title from stack | FS-VT-061 |

OSC 22/23 are xterm title stack extensions (SHOULD, not Must). Stack depth is not mandated; a depth of 10 is sufficient.

#### Sanitization rules (FS-VT-062, ARCHITECTURE.md §8.1)

1. Strip all C0 control characters (0x00–0x1F) and C1 control characters (0x80–0x9F)
2. Truncate to 256 characters after stripping

#### Read-back prevention (FS-VT-063)

`CSI 21 t` (report window title) MUST produce no response. Zero bytes written to PTY. This prevents the title from being injected into the shell's stdin via a read-back query.

#### Propagation path (ARCHITECTURE.md §4.3, §4.5.2)

OSC → `VtProcessor::osc_dispatch()` → update `ScreenBuffer.title` → emit `session-state-changed` with `changeType: 'pane-metadata-changed'` → frontend updates `PaneState.title` in its `TabState` replica → tab header re-renders.

There is no dedicated `title-changed` event. The title travels inside the `TabState` embedded in `session-state-changed`.

#### `VtTestCase` fixtures

| Test name | Input | Expected |
|---|---|---|
| `FS-VT-060-osc0-bel` | `\x1b]0;Hello\x07` | `title = "Hello"` |
| `FS-VT-060-osc2-st` | `\x1b]2;World\x1b\\` | `title = "World"` |
| `FS-VT-060-osc1-ignored` | `\x1b]1;IconName\x07` | `title` unchanged (or set, but no crash) |
| `FS-VT-062-strip-c0` | `\x1b]0;Ti\x01tle\x07` | `title = "Title"` (0x01 stripped) |
| `FS-VT-062-strip-sgr` | `\x1b]0;Ti\x1b[31mtle\x07` | `title = "Ti[31mtle"` or `"Title"` (ESC stripped) |
| `FS-VT-062-truncate` | `\x1b]0;` + `A`×300 + `\x07` | `title.len() == 256` |
| `FS-VT-063-csi21t-no-response` | `\x1b[21t` | zero bytes written to PTY input |
| `FS-VT-060-osc22-push-pop` | `\x1b]22;First\x07` then `\x1b]2;Second\x07` then `\x1b]23;\x07` | `title = "First"` after pop |

For `FS-VT-062-strip-sgr`: the ESC byte (0x1B) is a C0 character and must be stripped. The remaining literal bytes `[31mtle` may be kept as printable ASCII or also stripped depending on strictness. The minimum acceptable behaviour is that the ESC byte does not survive into the displayed title, ensuring no colour injection into the tab bar.

### 1.4 Focus Events Mode 1004 (FS-VT-084)

#### Standard reference

xterm `ctlseqs.ms` §Focus Tracking. This is an xterm extension; not in ECMA-48.

#### Sequences

```
enable:     ESC [ ? 1004 h
disable:    ESC [ ? 1004 l
focus-in:   ESC [ I   (sent to PTY when pane gains focus)
focus-out:  ESC [ O   (sent to PTY when pane loses focus)
```

Note: `ESC [ O` is also the application cursor mode sequence for F4 on some terminals. In practice, applications that use focus events check for mode 1004 explicitly and do not confuse focus-out with F4.

#### Frontend responsibility

The `VtProcessor` handles DECSET/DECRST 1004 and updates mode state. The frontend must:
1. Receive `mode-state-changed` with `focusEvents: bool`
2. Register `focus`/`blur` event listeners on the pane element
3. When `focusEvents === true`: on `focus`, call `send_input({ paneId, data: [0x1B, 0x5B, 0x49] })`; on `blur`, call `send_input({ paneId, data: [0x1B, 0x5B, 0x4F] })`
4. When `focusEvents === false`: do nothing on focus/blur

The mode-state-changed payload must include `focusEvents: bool`. This is currently absent from the ARCHITECTURE.md §4.3 definition and must be added.

#### Application compatibility

| Application | Uses focus events | Why |
|---|---|---|
| vim/neovim | Yes (`autoread`, `CursorHold` reset) | Detect external file changes; battery conservation |
| tmux | Yes | Session activity tracking |
| emacs | Yes (with appropriate config) | Buffer refresh on focus |
| kakoune | Yes | External modification detection |

#### `VtTestCase` fixtures (mode-state only; byte emission is frontend-side)

| Test name | Input | Expected mode |
|---|---|---|
| `FS-VT-084-decset-1004` | `\x1b[?1004h` | `focusEvents: true` |
| `FS-VT-084-decrst-1004` | `\x1b[?1004h` then `\x1b[?1004l` | `focusEvents: false` |
| `FS-VT-084-reset-on-session-close` | `\x1b[?1004h` then session close | `focusEvents: false` |

### 1.5 DECKPAM / DECKPNM (FS-KBD-010)

#### Standard reference

DEC VT220 Programmer Reference Manual §4.6.1 (DECKPAM/DECKPNM). ECMA-48 §8.3.48.

#### Sequences

```
DECKPAM: ESC =   (0x1B 0x3D) — keypad application mode
DECKPNM: ESC >   (0x1B 0x3E) — keypad numeric mode (default)
```

These are not DECSET/DECRST sequences — they are two-character Esc sequences with no CSI prefix.

#### Application keypad sequences (SS3 — Single Shift 3)

In application mode (DECKPAM active), keypad keys emit SS3 sequences:

```
ESC O <char>   (0x1B 0x4F <char>)
```

| Key | Numeric mode | Application mode |
|---|---|---|
| KP_0 | `0` (0x30) | `ESC O p` (0x1B 0x4F 0x70) |
| KP_1 | `1` (0x31) | `ESC O q` (0x1B 0x4F 0x71) |
| KP_2 | `2` (0x32) | `ESC O r` (0x1B 0x4F 0x72) |
| KP_3 | `3` (0x33) | `ESC O s` (0x1B 0x4F 0x73) |
| KP_4 | `4` (0x34) | `ESC O t` (0x1B 0x4F 0x74) |
| KP_5 | `5` (0x35) | `ESC O u` (0x1B 0x4F 0x75) |
| KP_6 | `6` (0x36) | `ESC O v` (0x1B 0x4F 0x76) |
| KP_7 | `7` (0x37) | `ESC O w` (0x1B 0x4F 0x77) |
| KP_8 | `8` (0x38) | `ESC O x` (0x1B 0x4F 0x78) |
| KP_9 | `9` (0x39) | `ESC O y` (0x1B 0x4F 0x79) |
| KP_Decimal | `.` (0x2E) | `ESC O n` (0x1B 0x4F 0x6E) |
| KP_Enter | CR (0x0D) | `ESC O M` (0x1B 0x4F 0x4D) |
| KP_Add | `+` (0x2B) | `ESC O k` (0x1B 0x4F 0x6B) |
| KP_Subtract | `-` (0x2D) | `ESC O m` (0x1B 0x4F 0x6D) |
| KP_Multiply | `*` (0x2A) | `ESC O j` (0x1B 0x4F 0x6A) |
| KP_Divide | `/` (0x2F) | `ESC O o` (0x1B 0x4F 0x6F) |

#### Frontend implementation constraint

The Web Platform delivers keypad events with `KeyboardEvent.code` values: `Numpad0`–`Numpad9`, `NumpadDecimal`, `NumpadEnter`, `NumpadAdd`, `NumpadSubtract`, `NumpadMultiply`, `NumpadDivide`.

`KeyboardEvent.key` is unreliable for this purpose: with NumLock on, `key === "5"` for both `Numpad5` and `Digit5`. The encoder MUST use `code` to identify numpad keys.

NumLock state: browsers do not expose a reliable NumLock API. When NumLock is OFF, many platforms report `key === "Clear"` for KP_5 or send arrow keys for KP_2/4/6/8. In application mode (DECKPAM), NumLock-off numpad keys should still emit the SS3 sequences (the application is expecting application-mode behavior regardless of NumLock). In numeric mode, NumLock-off numpad keys may emit arrow/navigation sequences — this is OS-dependent and can be documented as a known platform limitation for v1.

#### Mode-state-changed propagation

The backend emits `mode-state-changed` with `deckpam: bool` when `ESC =` or `ESC >` is received by the VtProcessor. The frontend `keyboard.ts` encoder must listen for this event per pane and use it to select the encoding table.

#### `VtTestCase` fixtures

| Test name | Input | Expected mode |
|---|---|---|
| `FS-KBD-010-deckpam` | `\x1b=` | `deckpam: true` |
| `FS-KBD-010-deckpnm` | `\x1b=` then `\x1b>` | `deckpam: false` |
| `FS-KBD-010-reset-on-session-close` | `\x1b=` then session close | `deckpam: false` |

The encoding of KP_5 → `ESC O u` is a frontend unit test (vitest), not a VtTestCase (the VtProcessor does not encode keyboard input — it only tracks mode).

## Part 2: SSH Domain — Protocol Constraints

### 2.1 SSH Authentication Sequence (FS-SSH-010 to 014)

#### Standard references

- RFC 4251 §6 (SSH Architecture)
- RFC 4252 (SSH Authentication Protocol) — `publickey`, `password`, `keyboard-interactive`
- RFC 4254 §6.2 (SSH Connection Protocol — channel requests: `pty-req`, `shell`)
- RFC 4254 Annex A (encoded terminal modes — opcode table)

#### Lifecycle state machine (ARCHITECTURE.md §5.2)

```
open_ssh_connection()
  → Connecting   (TCP in progress)
  → Authenticating  (SSH handshake + credential exchange)
  → Connected    (interactive PTY)
  ↓ network drop / keepalive timeout / non-zero exit
  → Disconnected   (reconnect available)
  ↓ user close / zero exit
  → Closed       (no reconnect available)
```

Key distinction: **Disconnected** (unexpected — reconnect available) vs **Closed** (deliberate or clean — no reconnect). The reconnect button MUST appear in Disconnected and MUST NOT appear in Closed.

#### Authentication order (FS-SSH-012)

Attempt order: `publickey` → `keyboard-interactive` → `password`. A saved connection may specify a preferred method; the order still applies as a fallback sequence.

`keyboard-interactive` is distinct from `password`: it is an interactive prompt sequence (RFC 4256) where the server may send multiple prompts. The frontend must handle multi-step keyboard-interactive auth, not just a single password prompt. `credential-prompt` events carry the prompt text from the server.

#### Terminal modes field in pty-req (FS-SSH-013, RFC 4254 Annex A)

The `pty-req` channel request payload includes an `encoded terminal modes` field. Each entry is a 5-byte tuple: `uint8 opcode` + `uint32 value` (big-endian). The list is terminated by `TTY_OP_END` (opcode 0).

**RFC 4254 Annex A opcode table (authoritative):**

| Opcode | Name | Value to send |
|---|---|---|
| 0 | TTY_OP_END | — (terminator) |
| 1 | VINTR | 3 (^C) |
| 2 | VQUIT | 28 (^\) |
| 3 | VERASE | 127 (DEL) |
| 4 | VEOF | 4 (^D) |
| 5 | VKILL | 21 (^U) |
| 6 | VEOL | 255 (unused/disabled) |
| 7 | VEOL2 | 255 |
| 8 | VSWTCH | 255 |
| 9 | VSTART | 17 (^Q) |
| 10 | VSUSP | 26 (^Z) |
| 50 | ISIG | 1 (enabled) |
| 51 | ICANON | 1 (enabled) |
| 53 | ECHO | 1 (enabled) |

**FS-SSH-013 erratum:** The FS.md documentation incorrectly assigns VKILL=4 and VEOF=5. The RFC 4254 values are VEOF=4, VKILL=5. The implementation is correct per RFC. The documentation error must not be "fixed" by inverting the implementation.

**Validation approach:** Parse the byte sequence of the `pty-req` terminal modes field byte by byte and assert each opcode/value pair. A unit test over the `build_pty_modes()` function (or equivalent) is the correct layer.

#### Host key TOFU (FS-SSH-011)

TauTerm maintains its own known-hosts file at `~/.config/tauterm/known_hosts`. It does NOT read `~/.ssh/known_hosts` at connection time (ARCHITECTURE.md §8.3). The preferences UI offers a manual "Import from OpenSSH" action.

First connection: `host-key-prompt` event carries `{ host, fingerprint_sha256, key_type, is_known: false, fingerprint_stored: null }`.

Key change: `host-key-prompt` event carries `{ is_known: true, fingerprint_stored: "<old>", fingerprint_sha256: "<new>" }`. The connection MUST be blocked. The frontend must not call `accept_host_key` without explicit user action — and Accept must be a non-default action.

Fingerprint format: SHA-256 base64-encoded, prefixed `SHA256:`, e.g., `SHA256:+DiY3wvvV6TuJJhbpZisF/zLDA0zPMSvHdkr4UvCOqU`.

### 2.2 SSH Reconnection (FS-SSH-040 to 042)

#### IPC command

`reconnect_ssh({ pane_id })` — defined in ARCHITECTURE.md §4.2. The command exists in the IPC contract; the UI that calls it does not yet exist.

#### Scrollback separator

On reconnection, the backend must insert a separator into the screen buffer before attaching the new PTY channel. The separator is a dedicated line type (not a regular text line) to ensure it survives search and scroll without being matched as regular content. Implementation options:
- A special `LineType::Separator { timestamp: String }` in the screen buffer
- A regular line with a unique attribute set (e.g., a distinct SGR color + text) that the frontend renders as a visual divider

The simpler option (regular line with distinct attributes) is acceptable for v1 if `LineType` is not yet modeled.

## Part 3: Search Domain (FS-SEARCH-001 to 007)

### 3.1 Architectural position

`search_pane` is a Tauri command that returns `Vec<SearchMatch>`. The VT processor must expose a search interface over the screen buffer. Per ARCHITECTURE.md §3.3, the `vt` module owns the screen buffer; search lives in `vt/search.rs`.

### 3.2 Soft-wrap handling

The screen buffer tracks a `soft_wrap: bool` flag per line (FS-SB-008). A search query must join soft-wrapped lines into logical lines before matching. The joining algorithm:

```
logical_line = ""
for each visual line L:
    logical_line += L.text
    if L.soft_wrap:
        continue  // don't break yet
    else:
        match against logical_line
        logical_line = ""
```

A match found in a logical line spanning visual lines M through N must be reported as a `SearchMatch` that carries `{ start_visual_line: M, start_col: c1, end_visual_line: N, end_col: c2 }` to allow the frontend to highlight across line boundaries.

### 3.3 Alternate screen exclusion (FS-SEARCH-004)

Search must only operate on the normal screen buffer's scrollback. When `ScreenBuffer.alternate_active == true`, `search_pane` returns `[]` immediately without scanning. No VtTestCase needed — this is a unit test on the `SearchEngine` struct.

### 3.4 Performance constraint (FS-SEARCH-005)

100,000 lines × ~80 characters = ~8 MB. At 100 ms budget:
- Naive regex compilation per line: ~100,000 compile cycles → fail
- Single-pass compiled regex over joined text: feasible
- Recommended: use the `regex` crate, compile once per query, apply `.find_iter()` over each logical line

For case-insensitive literal matching without regex, `memchr` + `std::str::to_lowercase` is sufficient and faster.

### 3.5 `SearchMatch` type

```rust
pub struct SearchMatch {
    pub logical_line: usize,        // logical line index in scrollback (after joining soft-wraps)
    pub start_visual_line: usize,   // first visual line of the match
    pub start_col: usize,           // column on start_visual_line
    pub end_visual_line: usize,     // last visual line of the match (may equal start)
    pub end_col: usize,             // column on end_visual_line (exclusive)
    pub matched_text: String,       // the actual matched bytes (for display / case-insensitive show)
```

## Part 4: IPC Wiring Constraints

### 4.1 mode-state-changed payload gap

Current definition (ARCHITECTURE.md §4.3):
```typescript
{ paneId: PaneId, decckm: bool, deckpam: bool }
```

Required additions for the 12 items:
```typescript
{ paneId: PaneId, decckm: bool, deckpam: bool, bracketedPaste: bool, focusEvents: bool }
```

Both fields are needed by the frontend to correctly handle FS-CLIP-008 and FS-VT-084. The Rust side (`events/types.rs`) and the TypeScript side (`src/lib/ipc/types.ts`) must be updated together. An `ipc_type_coherence.rs` integration test should assert the JSON shape of the extended payload.

### 4.2 Notification-changed event routing

`notification-changed` event (ARCHITECTURE.md §4.5.4):
```typescript
interface NotificationChangedEvent {
  tabId: string;
  paneId: string;
  notification: PaneNotification | null;
}
```

The frontend must:
1. Register a `listen('notification-changed', handler)` in the session state module
2. On receipt, traverse the `TabState.layout` tree to find the `PaneNode::Leaf` with `paneId`
3. Update `PaneState.notification` in the frontend replica
4. Trigger re-render of the tab header (which reads `notification` from the active `PaneState`)

The `null` value means "clear notification" — the frontend must render no indicator when `notification === null`.

### 4.3 set_active_pane and session-state-changed

Per ARCHITECTURE.md §4.5.2, `session-state-changed` with `changeType: 'active-pane-changed'` is emitted by the backend when `set_active_pane` is processed. The frontend must not optimistically update `activePaneId` before receiving this event — it should use the event as the authoritative state update to avoid divergence between frontend and backend active pane tracking.

In practice, for responsiveness, an optimistic update is acceptable if the frontend immediately reconciles on receiving the event. The test must verify that the final state after the event is consistent with what the backend reports.

### 4.4 OSC title and session-state-changed timing

OSC title changes originate asynchronously from PTY output. The `session-state-changed` event is emitted from the PTY read task (ARCHITECTURE.md §6.2). The frontend must not assume that the title in the current `TabState` snapshot (from `get_session_state`) is up to date — it must apply delta updates from `session-state-changed` events on top of the snapshot.

### 4.5 Credential injection in ssh_cmds.rs

`CredentialManager` (or `CredentialStore`) must be injected into Tauri's managed state and accessed in `ssh_cmds.rs` via `State<CredentialStore>`. The pattern follows `State<PreferencesStore>` already in use. The handler calls:

```rust
async fn open_ssh_connection(
    state: State<'_, SessionRegistry>,
    cred_store: State<'_, Arc<dyn CredentialStore>>,
    prefs: State<'_, PreferencesStore>,
    pane_id: PaneId,
    connection_id: ConnectionId,
) -> Result<(), TauTermError>
```

If `cred_store.is_available()` returns false (no Secret Service), the handler must not attempt to retrieve credentials — instead it must emit a `credential-prompt` event immediately with an `unavailable: true` flag so the frontend can display the "keychain unavailable" notice.
